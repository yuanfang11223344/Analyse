# Git分支-变基

在 Git 中整合来自不同分支的修改主要有两种方法：`merge` 以及 `rebase`。本节将学习什么是"变基"，怎样使用它，展示该操作的惊艳之处，以及指出在何种情况下应避免使用。

## 变基的基本操作

回顾分支合并的例子，开发任务分叉到两个不同分支，又各自提交了更新。整合分支最容易的方法是 `merge` 命令，它会对两个分支的最新快照（`C3` 和 `C4`）以及二者最近的共同祖先（`C2`）进行三方合并，生成一个新的快照并提交。

### 概念

**"变基就是：将某一分支上的所有修改复制到另一分支上"**

除了 `merge`，还可以提取在 `C4` 中引入的补丁和修改，然后在 `C3` 的基础上应用。在 Git 中，这种操作叫做**变基（rebase）**。你可以使用 `rebase` 命令将提交到某一分支上的所有修改都移到另一分支上，仿佛"重新播放"一样。

在此例中，可以检出 `experiment` 分支，然后将它变基到 `master` 分支上：

```shell
$ git checkout experiment
$ git rebase master # 将experiment上的修改变基到master分支上（将experiment的提交移动到master上。）
First, rewinding head to replay your work on top of it...
Applying: added staged command
```

### 原理

1. **找到当前分支和目标分支的最近共同祖先**
2. **对比当前分支相对于该共同祖先的历次提交**
3. **提取相应的修改并存为临时文件**
4. **将当前分支指向目标分支**
5. **将之前临时文件的修改依序应用**

现在回到 `master` 分支，进行一次快进合并：

```shell
$ git checkout master
$ git merge experiment
```

### 步骤

**先检出源分支，将源分支的修改变基到目标分支。切回目标分支，进行一次快进合并**

```shell
# 示意：
git checkout <源分支>
git (源分支的修改)rebase(到) <目标分支>
git checkout <目标分支>
git merge <源分支>
```

此时，`C4'` 指向的快照就与 merge 示例中 `C5` 指向的快照一模一样。这两种整合方法的最终结果没有区别，但**变基使得提交历史更加整洁**。查看经过变基的分支历史记录时会发现，尽管实际的开发工作是并行的，但它们看上去像串行一样，提交历史是一条直线没有分叉。

这样做通常是为了向远程分支推送时保持提交历史的整洁——例如向他人维护的项目贡献代码时，先在自己的分支里开发，完成后先将代码变基到 `origin/master` 上，再向主项目提交修改。这样项目维护者就不需要再进行整合工作，只需快进合并即可。

请注意，无论是通过变基还是通过三方合并，整合的最终结果所指向的快照始终一样，只是提交历史不同。变基是将一系列提交按照原有次序依次应用到另一分支上，而合并是把最终结果合在一起。

### 优点

**变基的优点：使提交记录更加整洁。**

## 更有趣的变基例子

在对两个分支进行变基时，所生成的"重放"并不一定要在目标分支上应用，你也可以指定另外的分支进行应用。例如创建了主题分支 `server`，为服务端添加功能，提交了 `C3` 和 `C4`。然后从 `C3` 上创建主题分支 `client`，为客户端添加功能，提交了 `C8` 和 `C9`。最后回到 `server` 分支，又提交了 `C10`。

假设希望将 `client` 中的修改合并到主分支并发布，但暂时不想合并 `server` 中的修改（还需更多测试）。这时可以使用 `git rebase` 命令的 **`--onto` 选项**，选中在 `client` 分支里但不在 `server` 分支里的修改（即 `C8` 和 `C9`），将它们在 `master` 分支上重放：

```shell
$ git rebase --onto master server client
```

以上命令的意思是："取出 `client` 分支，找出它从 `server` 分支分歧之后的补丁，然后把这些补丁在 `master` 分支上重放一遍，让 `client` 看起来像直接基于 `master` 修改一样"。

### --onto选项

**选中C分支中的但不在B分支里的修改，应用到A分支。**

现在可以快进合并 `master` 分支了：

```shell
$ git checkout master
$ git merge client
```

### 省去先切换到源分支的步骤

```shell
git rebase <目标(当前)分支> <源分支> # 将源分支变基到目标分支。执行此命令后会自动切换到源分支
git checkout <目标分支>
git merge <源分支>
```

**注意：使用这个方法要确保源分支上的代码是最新的。**

接下来将 `server` 分支中的修改也整合进来。使用 `git rebase <basebranch> <topicbranch>` 命令可以直接将主题分支（即本例中的 `server`）变基到目标分支（即 `master`）上，省去先切换到 `server` 分支再执行变基的多个步骤：

```shell
$ git rebase master server
```

然后快进合并主分支 `master`：

```shell
$ git checkout master
$ git merge server
```

至此，`client` 和 `server` 分支中的修改都已整合到主分支，可以删除这两个分支：

```shell
$ git branch -d client
$ git branch -d server
```

## 变基的风险

### 金科玉律

变基并非完美无缺，需遵守一条准则：

**如果提交存在于你的仓库之外，而别人可能基于这些提交进行开发，那么不要执行变基。**

遵循这条金科玉律就不会出差错。

> "例如：几个人同时在一个主题分支上进行开发和提交时，你不要中途执行变基，只有在大家都完成工作之后才可以执行变基。"

### 变基的实质

"变基操作的实质是丢弃一些现有的提交，然后相应地新建一些内容一样但实际上不同的提交。" 如果已将提交推送至某个仓库，而他人也已从该仓库拉取提交并进行了后续工作，此时若用 `git rebase` 重新整理了提交并再次推送，同伴将不得不再次整合他们的工作与你的提交，若接下来还要拉取并整合他们修改过的提交，事情会变得一团糟。

来看一个在公开仓库上执行变基带来的问题。假设从中央服务器克隆然后在它的基础上进行开发。然后，某人向中央服务器提交了一些修改，其中包括一次合并。你抓取了这些远程分支上的修改，并将其合并到本地开发分支。

接下来，这个人决定把合并操作回滚，改用变基；又用 `git push --force` 覆盖了服务器上的提交历史。之后从服务器抓取更新，会发现多出一些新的提交。结果两人处境都十分尴尬。如果执行 `git pull`，将合并来自两条提交历史的内容，生成一个新的合并提交。

此时若执行 `git log`，会发现有两个提交的作者、日期、日志居然一样，令人混乱。此外，若将这一堆推送到服务器上，实际上是把那些已被变基抛弃的提交又找了回来，更加混乱。

## 用变基解决变基

如果真遭遇了类似处境，Git 还有一些高级魔法可以帮忙。需要做的是检查做了哪些修改，以及对方覆盖了哪些修改。

实际上，Git 除了对整个提交计算 SHA-1 校验和以外，也对本次提交所引入的修改计算了校验和——即 "patch-id"。

如果拉取被覆盖过的更新并将手头的工作基于此进行变基，一般 Git 都能成功分辨出哪些是你的修改，并把它们应用到新分支上。

举例来说，如果遇到前述那种情境，不执行合并，而是执行 `git rebase teamone/master`，Git 将会：

- 检查哪些提交是我们分支上独有的（C2，C3，C4，C6，C7）
- 检查其中哪些提交不是合并操作的结果（C2，C3，C4）
- 检查哪些提交在对方覆盖更新时并没有被纳入目标分支（只有 C2 和 C3，因为 C4 其实就是 C4'）
- 把查到的这些提交应用在 `teamone/master` 上面

要使上述方案有效，还需要对方在变基时确保 `C4'` 和 `C4` 是几乎一样的。否则变基操作将无法识别，并新建另一个类似 `C4` 的补丁（这个补丁很可能无法整洁地整合入历史，因为补丁中的修改已存在于某个地方了）。

在本例中另一种简单的方法是使用 `git pull --rebase` 命令而不是直接 `git pull`。又或者可以手动完成：先 `git fetch`，再 `git rebase teamone/master`。

如果习惯使用 `git pull`，又希望默认使用 `--rebase` 选项，可执行：

```shell
$ git config --global pull.rebase true
```

来更改 `pull.rebase` 的默认配置。

如果只对不会离开电脑的提交执行变基，那就不会有事。如果对已推送过的提交执行变基，但别人没有基于它的提交，也不会有事。如果对已推送至共用仓库的提交执行变基，并因此丢失了一些别人的开发所基于的提交，那就麻烦了。

如果决意要这么做，请务必通知每个人执行 `git pull --rebase` 命令，这样尽管不能避免伤痛，但能有所缓解。

## 变基 vs. 合并

至此已学习了变基和合并的用法，自然会问到底哪种方式更好。先退一步讨论提交历史到底意味着什么。

一种观点认为，仓库的提交历史是**记录实际发生过什么**。它是针对历史的文档，本身就有价值，不能乱改。从这个角度看，改变提交历史是一种亵渎——用谎言掩盖了实际发生过的事情。即便合并产生的提交历史一团糟，既然事实如此，这些痕迹就应该保留下来，让后人查阅。

另一种观点认为，提交历史是**项目过程中发生的事**。没人会出版一本书的第一版草稿，软件维护手册也需要反复修订才能方便使用。持这一观点的人会使用 `rebase` 及 `filter-branch` 等工具来编写故事，怎么方便后来的读者就怎么写。

现在回到问题：到底合并还是变基好？这并没有一个简单的答案。Git 是非常强大的工具，允许对提交历史做许多事情，但每个团队、每个项目的需求并不相同。既然已分别学习了两者的用法，相信能根据实际情况作出明智选择。

**总的原则是，只对尚未推送或分享给别人的本地修改执行变基操作来清理历史，从不对已推送至别处的提交执行变基操作，这样才能享受到两种方式带来的便利。**
